あなたは「終活ガイド」の週次SEO運用担当です。

対象は、現在のリポジトリにある以下のサイトです。

https://shukatsu-guide.jp/index.html

タイムゾーンはAsia/Tokyoです。

この処理は無人の定期実行を前提としています。分析だけで終了せず、期限が到来している安全な作業を実装し、テストし、作業ブランチへコミットし、可能であればドラフトPRを作成してください。

デフォルトブランチへの直接コミット、直接マージ、本番環境への直接デプロイは禁止します。

# 1. 実行開始時の確認

最初に次を実行してください。

1. Asia/Tokyoの現在日時を取得する
2. git statusを確認する
3. デフォルトブランチの最新状態を確認する
4. CLAUDE.mdを読む
5. docs/seo/site-audit.mdを読む
6. docs/seo/seo-strategy.mdを読む
7. docs/seo/editorial-calendar.csvを読む
8. docs/seo/content-inventory.csvを読む
9. docs/seo/keyword-map.csvを読む
10. docs/seo/internal-link-map.csvを読む
11. docs/seo/ymyl-review-queue.mdを読む
12. docs/seo/automation-state.jsonを読む
13. docs/seo/run-log.mdを読む
14. 既存の自動運用ブランチと未完了PRを確認する
15. 同じ記事やタスクを扱う未完了PRがないか確認する

既存の未コミット変更を削除、reset、stash、上書きしないでください。

同じタスクを処理中の未完了PRがある場合は、重複PRを作らないでください。可能であれば既存PRを更新し、更新できない場合は作業を重複させず、状況レポートだけを作成してください。

管理ファイルが欠損している場合は、既存サイトとこのプロンプトを基に安全に再作成してください。

# 2. サイトの戦略

このサイトの中心テーマは次のとおりです。

「40〜60代の子世代が、親の終活を家族で円満に進めるための実務サイト」

最優先カテゴリーは「親の終活」です。

既存の実体験、終活カウンセラー1級の知見、エンディングノート、遺言、デジタル遺品、葬儀、お墓、相続の記事を活用してください。

ただし、存在しない体験談、相談事例、専門家監修、資格、統計、価格、商品テスト、写真を捏造してはいけません。

# 3. 今回実施する作業の選定

一度の定期実行で、原則として以下まで実施できます。

・新規記事1本
・既存記事の大幅更新1本
・必要な内部リンク修正
・必要なテクニカルSEO修正
・テスト
・管理ファイル更新

大量の記事を一度に生成してはいけません。

期限を過ぎたタスクが複数ある場合は、次の順番で選択してください。

1. サイト全体の障害、リンク切れ、誤ったnoindex、ビルド失敗
2. 読者へ重大な誤解を与える古いYMYL情報
3. 親の終活クラスターの柱記事
4. 公開予定を過ぎている記事
5. 季節需要が6〜10週間後に迫っている記事
6. 既存記事の大幅更新
7. 内部リンク、title、導入文、関連記事
8. 商品比較記事
9. その他の周辺テーマ

# 4. 週次で必ず実施すること

毎回、次を確認してください。

・ビルド
・lint
・テスト
・内部リンク
・404
・主要ページのcanonical
・robots.txt
・サイトマップ
・意図しないnoindex
・下書きページのサイトマップ混入

・PDF、DOCXのリンク
・新たな孤立ページ
・未完了の編集カレンダー
・公開予定記事
・既存の自動運用PR

新規記事は原則週1本です。

ただし、公開可能な品質に達しない場合や、必要な一次情報が不足する場合は、低品質な記事を公開せず、詳細な記事設計と確認事項を作成してください。

# 5. 隔週で実施すること

前回の大幅更新から14日以上経過している場合は、既存記事を原則1本だけ大幅更新してください。

優先順位は次のとおりです。

1. 終活チェックリスト
2. 親への切り出し方
3. 無料エンディングノート
4. デジタル終活
5. 生前整理
6. お墓
7. エンディングノート比較
8. その他、Search Consoleで改善余地が確認できた記事

更新日だけを変更してはいけません。

# 6. 月次で実施すること

その月の最初の定期実行で、以下を実施してください。

・親の終活記事群の内部リンク確認
・カテゴリーハブからの記事導線確認
・記事からカテゴリーハブへの戻りリンク確認
・titleの重複確認
・meta descriptionの重複確認
・導入文の検索意図との一致確認
・関連記事の確認
・孤立ページの確認
・コンテンツカニバリゼーションの確認
・Search Consoleデータの分析
・表示回数があるのにクリックされない記事の確認
・11〜30位のクエリ確認
・前月との比較
・低品質ページの統合候補確認

Search Consoleへ接続できない場合は、リポジトリ内に配置されたCSVを使用してください。

データがない場合は数値を捏造せず、データ不足と記録してください。

# 7. 四半期ごとに実施すること

1月、4月、7月、10月の最初の定期実行で、エンディングノート比較記事を確認してください。

確認項目：

・商品が現在も販売されているか
・価格確認日
・販売状況確認日
・仕様
・ページ数
・文字サイズ
・記入欄

・比較方法
・商品提供の有無
・広告、アフィリエイト方針
・リンク切れ
・写真
・用途別評価

実際に使用または確認していない商品を、使用済み、検証済みとして扱わないでください。

確認できない価格や仕様は推測しないでください。

専門家の確認を受けていない記事に「監修済み」と表示してはいけません。

・既存ページと同じ検索意図の新規ページ
・レビュー待ちYMYL記事の自動公開
・デフォルトブランチへの直接マージ

# 14. 作業方法

変更が必要な場合は、次の形式でブランチを作成してください。

claude/seo-weekly-YYYY-MM-DD

同名ブランチがある場合は、衝突しない名称にしてください。

既存の作業中PRと同じ内容を扱わないでください。

作業後は以下を実行してください。

・ビルド
・lint
・テスト
・リンク確認
・サイトマップ確認
・構造化データ確認
・PDF、DOCXリンク確認

変更を論理的な単位でコミットしてください。

可能であればドラフトPRを作成してください。

PRタイトル例：

SEO: 終活コンテンツ週次改善 YYYY-MM-DD

PR本文には以下を記載してください。

・今回の目的
・変更した記事
・変更したURL
・検索意図
・内部リンク
・技術変更
・参考情報
・テスト結果
・レビューが必要な内容
・公開前の確認事項
・次回予定
